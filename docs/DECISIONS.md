## 2026-07-12 — Financial reports: Trial Balance, Income & Expense Statement, Balance Sheet
**Decision**: Added `/accounting/reports` with three reports, explicitly labeled with their real limitations rather than implying textbook accounting rigor the schema can't back up:
- **Trial Balance** — cash/bank account balances as of a chosen date (reuses the `balanceAsOf()` math already proven correct in the Cash/Bank Book work). Labeled plainly as an account-balances summary, not a full double-entry trial balance, since this schema has no income/expense/receivable *accounts* to list alongside cash/bank.
- **Income & Expense Statement (P&L)** — income and expenses grouped by category over a date range, with net surplus/deficit. Maps directly onto existing `income`/`expenses` tables, no caveats needed here.
- **Balance Sheet** — Assets = cash/bank balances + outstanding maintenance dues (as Accounts Receivable). Liabilities are always ₹0 (no loans/payables table exists anywhere in this schema) so Accumulated Fund (Equity) equals total Assets by construction — labeled explicitly as balancing by construction, not as an independent check, and as a snapshot of today rather than a historical as-of-date statement (bill balances aren't tracked with historical snapshots).
**Reason**: Resolved via a design discussion before coding, given the alternative (a real chart-of-accounts with account types and rewired posting across `BillingService`/`AccountingService`) would have meant reworking an already-built, working accounting module for a T3-scale change. User chose extending the existing single-entry cash-ledger model with clearly labeled reports instead — same reports the spec named, built honestly against what the data actually supports rather than overclaiming GAAP correctness.
**Impact**: No schema change. New methods: `Account::balanceAsOf()` (inclusive-of-date variant of the existing `balanceBefore()`), `Income::summaryByCategory()`, `Expense::summaryByCategory()`, `MaintenanceBill::totalOutstanding()`. Four new views, three new `AccountingController` actions plus a reports index, four new routes, all gated by the existing `accounting.view` permission — no new permission key needed.
**Verified end-to-end** against the live server with real transactions: set a cash account's opening balance to ₹2,000, recorded two real income entries (₹1,200 Maintenance, ₹300 Parking) and one real expense (₹400 Repairs) through the actual UI. ## 2026-07-12 — Backup & Restore, hard-restricted to super_admin
**Decision**: New `/admin/backup` page: create a full DB backup on demand, see a history of backups on the server, download any of them, delete old ones, and restore the database from either a stored backup or a freshly uploaded `.sql` file. Resolved via a requirements gate + design discussion first, given this is the single most destructive action in the app:
- **Dump mechanism**: pure PHP/PDO (`SHOW TABLES` → `SHOW CREATE TABLE` → `SELECT *` per table, values escaped via `PDO::quote()`), not `shell_exec('mysqldump ...')` — this app had zero shell-exec usage anywhere before this feature, and staying that way avoids a new class of attack surface (shell arg injection) and a new environment dependency (mysqldump in PATH).
- **Statement delimiting**: standard SQL uses `;` to end a statement, but a value can legitimately *contain* a `;` (or a literal newline) — parsing that back out reliably would need a real SQL tokenizer. Sidestepped entirely: the dump writer inserts a unique sentinel line (`-- ###SOCIETYOS_STMT_END###`) between statements, and restore just explodes on that sentinel. Known limitation, accepted rather than over-built: a text field whose value contains a raw newline immediately followed by `--` at the very start of the next line could have that continuation line mistaken for a SQL comment during restore. Rare in this app's actual data (addresses/descriptions), not worth a full parser.
- **Retention**: backups persist in `storage/backups/` (outside web root, gitignored, same "never linked directly" pattern as staff documents) with a history list, not stream-only.
- **Pre-restore safety net**: every restore — from a stored backup or an upload — automatically takes a fresh backup of the *current* state immediately before touching anything, and reports its filename in the success/error message either way. MySQL implicitly commits on DDL (`DROP`/`CREATE TABLE`), so a restore can't be wrapped in one atomic transaction; this safety backup is what actually makes a bad restore recoverable, not transactional rollback.
- **Access**: hard-restricted to `Auth::role() === 'super_admin'`, checked directly in `BackupController`, not merely gated behind a grantable `{module}.manage` permission key — so it can never end up in a lower-privilege role's permission set by accident the way other admin features can.
**Impact**: New `App\Services\BackupService` (dump/restore), `App\Controllers\BackupController`, `app/Views/admin/backup.php`, six new routes under `/admin/backup`, `storage/backups/` added to `.gitignore`. Filenames are validated against a strict pattern the controller itself generates (`resolveBackupPath()`) before any filesystem call — no path traversal surface even though the filename arrives via the URL. Restoring an older backup can leave the current session referencing a user/data that no longer exists post-restore; the UI flash message advises logging out and back in, and the post-restore activity-log write is wrapped in try/catch so a stale session doesn't turn a successful restore into an error page.
**Verified end-to-end** against the live DB, with an independent `mysqldump` snapshot taken outside the app first as an extra safety net before testing restore against real data: created a real backup via the UI, confirmed the file's structure (56 `DROP TABLE`, 186 `INSERT` statements matching the live table/row counts) and confirmed `/admin/backup/{file}/download` serves back a byte-identical copy. Captured row counts across six tables (members, bills, users, activity_logs, penalties, society), then restored from that same backup (a same-state round-trip — the safest possible live test) and confirmed every count matched afterward, confirmed the automatic pre-restore safety backup file existed on disk, and confirmed the activity log itself reflected the exact sub-second timing of the restore (a `backup_create` log entry present before the restore was correctly absent afterward, replaced by the `restore` entry — proof the restore reverted to the precise pre-backup_create moment, not an approximation). Tested path traversal (`../`) on the download route — 404, blocked. Created a temporary non-`super_admin` test user, confirmed `/admin/backup` returned 403 for it, then deleted that test user and its login/activity history afterward. All test backup files removed after verification; final DB state matched the pre-test baseline exactly.

## 2026-07-12 — Persisted billing penalty/interest, using the previously-unused `penalties` table
**Decision**: This was explicitly deferred earlier this session (see the "Phase 3" entry below) pending a policy decision. Resolved via a requirements gate + design discussion before coding:
- **Accrual trigger**: recomputed on view (bill detail page, defaulter report), not a scheduled job — the app has no job-runner infra, and this matches how `status='overdue'` detection already works (computed at query time).
- **Interest base**: current outstanding balance (`total_amount - paid_amount`), not the original total — pay down the bill, interest owed going forward drops.
- **Placement**: kept as a separate add-on (`penalties` table, unmerged into `total_amount`/`paid_amount`) rather than folded into the bill — the base bill's total stays a stable historical record; the payment form only ever settles the base bill.
- **Rate/grace**: single config constant `penalty_interest_rate_percent` (default 18% p.a., `.env`-overridable), no grace period — interest starts the day after `due_date`.
- **Formula**: `penalty = days_overdue × (rate/100/365) × current_outstanding_balance`, recomputed fresh on every view (upsert, not accumulate) — a running estimate against today's balance, not a day-by-day ledger across multiple partial payments. Flagged this simplification to the user explicitly before building, since it was the reason the feature was deferred the first time.
- **Once fully paid**: recalculation stops once outstanding balance hits ₹0 — the last computed penalty is left in place (still shown as owed) rather than reset to zero, since the late fee itself hasn't been collected just because the base bill has been.
**Impact**: New `App\Models\Penalty` (find-by-bill, upsert) and `App\Services\PenaltyService::recalculate()`. Wired into `BillingController::show()` and both defaulter-report code paths (`BillingController::defaulters()` and `ReportController::defaulters()`, which share the same view). Bill detail page shows a "Late payment interest" banner when penalty > 0, with an explicit note that it's charged separately. Defaulter report gained Penalty and Total Due columns (also in the CSV export). No schema change — `penalties` table existed since Phase 1.
**Verified end-to-end** against the live DB with a real overdue seed bill (2 days overdue, ₹2,800 outstanding, 18% p.a.): hand-calculated expected penalty (₹2.76) and confirmed the bill detail page, the `penalties` table row, and the defaulter report's Penalty/Total Due columns all matched exactly. Viewed the bill twice more and confirmed the `penalties` table still held exactly one row (upsert, not duplicate rows per view). Recorded a full payment and confirmed the bill flipped to `status='paid'` while the penalty amount stayed frozen at ₹2.76 rather than resetting to ₹0. Test payment/receipt/penalty rows reverted afterward — bill #1 restored to its original `unpaid`/₹0.00 seed state.

**Security design, since this is the app's first file upload feature**: built `App\Helpers\FileUpload` to (1) validate the actual file content via `finfo_file()`, never the client-supplied MIME type or original filename/extension — both are attacker-controlled; (2) generate a random filename, discarding the original entirely; (3) use `move_uploaded_file()`, which only succeeds on a file that genuinely arrived via HTTP upload; (4) enforce the existing `UPLOAD_MAX_SIZE_MB` config. Files are stored under `uploads/` (outside the web root, per the Phase 1 folder layout) and are **never linked to directly** — a new `StaffController::serveFile()` action streams them only after the route's normal auth + `staff.manage` permission middleware has already run, so there's no way to reach a staff photo, ID proof, or police verification certificate without being logged in with the right role.

## 2026-07-12 — Family member age: computed live from DOB, never stored stale
**Decision**: Discussed with user first. Added a nullable `date_of_birth` column to `family_members` alongside the existing `age` column. ## 2026-07-12 — Parking charges: dedicated effective-dated rate table, bundled into the existing bill, no proration
**Decision**: Discussed with user first (per this repo's process). Added billable parking charges: a new `parking_rates` table (slot_type, amount, effective_from — same effective-dating pattern as `maintenance_head_rates`, kept as a separate table rather than reusing `maintenance_heads`, since a parking charge is a per-allocated-slot count multiplied by a type-specific rate, not a per-flat attribute). Charges are added as extra line items on the flat's existing monthly bill (not a separate invoice), with **no proration** — an allocation that overlaps the billing period at all gets the full-period charge, same simplicity as maintenance heads.
**Reason for the schema change this required**: `bill_items.maintenance_head_id` was `NOT NULL`, so a parking-sourced line item literally couldn't be inserted. Loosened it to nullable, added a nullable `parking_slot_id`, and added a `CHECK` constraint (`chk_bill_item_source`) enforcing exactly one of the two is ever set — so a bill item's source is unambiguous and DB-enforced, not just a convention. `MaintenanceBill::items()` no longer JOINs to `maintenance_heads` for the display label; it now reads the `description` column set at generation time. This is more correct, not just a workaround: it means a bill's line-item text reflects what was true *when the bill was generated*, not whatever the head/slot is named today (or whether it still exists at all).
**Billing generation logic**: for each flat, in addition to the existing maintenance-head loop, `BillingService::generateForPeriod` now calls `ParkingAllocation::activeForFlatDuringPeriod()` (allocations overlapping `period_start`..`period_end`) and looks up `ParkingRate::amountAsOf(societyId, slotType, periodStart)` per allocation — same "rate as of the period's start date, not today" rule as maintenance heads, so a scheduled rate change bills correctly for both past and future periods without special-casing. No rate configured for a slot type yet = that allocation isn't charged (parking stays free until priced, same as an un-rated maintenance head).
**A real behavior change, called out explicitly**: previously, a flat with zero billable items (e.g. no maintenance heads had an effective rate yet) would still get a ₹0 bill created. Now such a flat is skipped entirely — a bill only gets created if there's at least one actual charge. `BillingService::generateForPeriod()`'s return signature grew a third counter (`noCharge`) alongside `created`/`skipped`, and `BillingController`'s flash message was updated so "skipped (already billed)" doesn't get conflated with "skipped (nothing to charge)" — those are different situations an admin would want to distinguish.
**Also fixed in passing**: found (via the same "would this crash" check applied to the Phase 12 category deletes) that deleting a `maintenance_head` or `parking_slot` that had ever been billed would throw an uncaught FK-violation 500, since neither delete action had a try/catch. **Verified end-to-end**: allocated a slot with no rate configured — confirmed it didn't appear on the generated bill. Configured a 4-wheeler rate effective a future date, generated a bill for a period *before* that date (parking correctly absent) and a period *after* it (parking correctly present, ₹250, bill total correctly summed to ₹3,050) — confirmed via direct `bill_items` inspection that `maintenance_head_id`/`parking_slot_id` are never both set, matching the CHECK constraint. ## 2026-07-12 — Maintenance head amounts: effective-dated rate history, not in-place edit
**Decision**: Discussed with user first (per this repo's process — design before code on anything schema-affecting). Replaced `maintenance_heads.amount` (a single mutable column) with a new `maintenance_head_rates` table: one row per rate, each with an `effective_from` date. A head's "current" amount is the row with the latest `effective_from <= today`; bill generation looks up the rate effective as of the *bill period's start date*, not "today" — so a rate scheduled in advance bills correctly for both past and future periods without any special-casing. Rates already in effect are append-only (no edit, no delete) — corrections are made by scheduling a new dated rate, matching how societies actually operate (a resolution passed with a future effective date). Only a not-yet-effective (future-dated) scheduled rate can be deleted, to fix a typo before it takes effect.
**Reason**: A plain in-place edit changes the amount immediately with no way to say "apply this from next month" and no record of what the old rate was outside of already-generated bills. This is a genuine T2/T3-tier schema change (new table, new FK, new lookup logic in bill generation), not a UI tweak — user confirmed the design before implementation began.
**Alternatives rejected**: keep the mutable column and just tell admins to edit on the day the change should apply — rejected because it can't represent an advance-scheduled change at all, which is exactly the real-world workflow (a passed resolution announced ahead of its effective date).
**Also checked and confirmed not needed elsewhere**: Staff Payroll (already dated per `pay_period`), Asset AMC (already has `start_date`/`end_date` per contract), Account opening balance (a one-time correction, not a recurring rate) — these already model time correctly by their existing structure, not by luck.
**Impact**: `database/schema.sql` — `maintenance_heads.amount` column removed, `maintenance_head_rates` table added (unique on `(maintenance_head_id, effective_from)`, `FOREIGN KEY ... ON DELETE CASCADE`). `database/seed.sql` updated to seed each demo head's initial rate through the new table. New `app/Models/MaintenanceHeadRate.php`; `MaintenanceHead::create()` now creates the head + its first rate in one transaction; `MaintenanceHead::update()` no longer touches amount. `BillingService::generateForPeriod()` now calls `MaintenanceHeadRate::amountAsOf($headId, $periodStart)` per head per bill and **skips** that line item entirely if no rate has taken effect yet by that period (rather than silently billing zero). New page `/society/maintenance-heads/{id}` for rate history + scheduling; the list page's inline edit now only covers name/type.
**Verified end-to-end**: scheduled a Sinking Fund increase (₹500→₹750) effective 2026-08-01 while "today" was 2026-07-12 — confirmed the list page still showed ₹500 with an "upcoming change" badge, generated a July bill (used ₹500, the old rate) and an August bill (used ₹750, the new rate) in the same test run, confirmed deleting the already-effective rate is blocked while deleting the future one succeeds, and confirmed scheduling a duplicate `effective_from` date is rejected with a friendly message instead of a raw DB error.

Persisting penalty amounts requires deciding *when* interest accrues (daily job? on view? on payment?) and how it composes with partial payments, which is a policy decision the spec doesn't pin down. Building it now risks guessing wrong and needing a rework.
**Alternatives rejected**: Compute-and-display-only penalty estimate inline (no DB write) — considered simpler, but decided to leave the whole feature for a dedicated pass rather than half-build it.
**Impact**: `penalties` table exists in schema but is unused. `maintenance_bills.status` ENUM already includes `'overdue'`; `MaintenanceBill::updatePaidAmountAndStatus()` sets it based on `due_date` at the moment a payment is recorded, but bills that go overdue without any payment activity keep `status='unpaid'` until touched — the defaulter report and `?status=` filter compensate by checking `due_date < CURDATE()` directly rather than trusting the stored status, so this doesn't affect correctness of user-facing reports, only the raw `status` column value. Verified end-to-end: bill generation (6 flats, 3 active heads → correct per-flat totals), partial payment → `partially_paid`, second payment → `paid`, PDF receipt downloads with valid `%PDF-1.7` header, defaulter report correctly surfaces both an aged batch and a newer overdue batch by due-date comparison.

## 2026-07-12 — Local dev/test access: port-based Apache vhost, not hosts-file hostname
**Decision**: Set up `http://localhost:8081/` as a dedicated Apache vhost (document root = `public/`) instead of a `societyos.local` hostname.
**Reason**: User asked to verify the app through real Apache/XAMPP, not just the PHP built-in dev server. A hostname-based vhost needs a `C:\Windows\System32\drivers\etc\hosts` entry, which requires admin rights this session doesn't have (attempted, got permission denied). **Impact**: `c:\xampp\apache\conf\extra\httpd-vhosts.conf` now has two blocks — a `ServerName localhost` vhost preserving existing `http://localhost/` behavior for other htdocs projects, and `Listen 8081` + a vhost on `*:8081` for SocietyOS. Apache was restarted to apply it. Verified: `localhost/` unaffected, `localhost:8081/` serves the landing page and full login→dashboard flow through mod_php.

## 2026-07-12 — Phase 5: Accounting, no auto-posting from billing payments
**Decision**: Built Accounts (cash/bank, computed running balance), Income, Expenses, Vendors, and a single Ledger view (filterable by account, doubling as cash book / bank book). Recording income or an expense auto-posts one ledger entry (credit or debit) in the same transaction. Maintenance bill payments (`BillingService::recordPayment`) do **not** post to the ledger — accounting and billing remain separate entry systems for now.
**Reason**: User was asked directly whether billing payments should auto-post to the ledger; asked to proceed with the simpler option. Auto-posting raises real questions this schema/spec doesn't answer yet — which account does a maintenance payment credit (there's no "linked collection account" concept on `maintenance_heads` or `society`), and does a *partial* payment post partially or wait for full settlement? Guessing either wrong means restating historical ledger entries later, which is worse than not having them yet.
**Alternatives rejected**: Auto-post every `payments` row to a hardcoded/default account — rejected because "default account" is itself a guess, and wrong ledger data is worse than a visible gap.
**Impact**: `ledger_entries.reference_type` ENUM already includes `'payment'` for when this is built — no schema change needed later, only a service-layer addition (`BillingService::recordPayment` would call something like `LedgerEntry::create(..., 'payment', $paymentId, ...)` once the target-account question is answered). Until then, a society using SocietyOS must record maintenance collections in Accounting → Income manually if they want it reflected in the ledger. This gap is visible in the UI (billing and accounting are separate menu items with no cross-link) rather than silently wrong.
**Also deferred**: Trial Balance / P&L / Balance Sheet reports — these require a proper chart-of-accounts (asset/liability/equity/income/expense classification) which `accounts`/`ledger_entries` as designed doesn't have (it's a single-entry cash-movement ledger per account, not double-entry bookkeeping). Building real financial statements on top of this schema would produce numbers that look authoritative but aren't accounting-correct; flagging rather than building a misleading report.
**Verified end-to-end** via the Apache vhost: created a bank account, recorded income (₹5,000) and an expense (₹1,800) against the seeded Cash account, confirmed the computed balance (0 opening + 5,000 − 1,800 = ₹3,200.00) matches, and confirmed the ledger's account filter correctly isolates entries per account (untouched Bank account correctly showed zero entries).

## 2026-07-12 — Phase 6: Visitors & Security, text token instead of scannable QR image
**Decision**: Built the gate register (log → approve/reject → checkout), pre-authorized visitor passes, and the delivery register. Passes use a random 8-char text token (`qr_token` column, still named per the existing schema) that gate staff type in to verify — not an actual scannable QR barcode image.
**Reason**: Rendering a real QR code needs either a JS library bundled locally (none in the project yet — CDN-only for Bootstrap/FontAwesome so far) or a server-side PHP QR library (not in `composer.json`, and DomPDF doesn't generate barcodes). Pulling in a new dependency for one feature warranted flagging rather than silently adding it. The token still gives the core value — a resident can pre-authorize a named visitor and the gate can check them in without a phone call — just via typed code instead of camera scan.
**Alternatives rejected**: Google Charts QR image API (deprecated/unreliable, and an external network dependency for something that should work on an offline LAN deployment) — rejected as a bad fit for a "single society, self-hosted" system.
**Impact**: `app/Models/VisitorPass.php` generates the token; `app/Views/visitors/passes.php` displays it as plain text for the resident to relay (verbally, WhatsApp, etc.) to their visitor. Upgrading to a real scannable QR later just means rendering `qr_token` through a QR image library in that same view — no schema or route change needed. Approving a visitor doesn't currently track *which* resident (member) approved it — `visitors.approved_by_member_id` is set to `NULL` always, since the logged-in `users` row isn't reliably linked to a `members` row yet (see the `users.member_id` column, unused so far).
**Verified end-to-end** via the Apache vhost: visitor log (pending) → approve → checkout; pass creation → verify/check-in (creates an approved visitor entry) → confirmed token reuse is blocked (DB row count stayed at 1); delivery log → mark collected.
